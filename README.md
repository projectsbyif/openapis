## What is this?

In collaboration with the Open Data Institute, IF researched how open APIs in the telecoms sector could lead to new classes of commercial products.

It is available at https://openapis.projectsbyif.com/.

This repo is a static site generator: Markdown files in `pages/` and settings in `settings.yaml` are compiled into static HTML in `public/`, with styles compiled from Sass and scripts bundled by [gulp](https://gulpjs.com/).

## Requirements

* **Node.js ≥ 18.** The version is pinned in `.nvmrc` (v20) — if you use [nvm](https://github.com/nvm-sh/nvm), run `nvm use` in the repo to match it.

## Installation

Clone and install:

```
git clone git@github.com:projectsbyif/openapis.git
cd openapis
npm install
```

`npm install` also runs `gulp deploy` automatically (via the `postinstall` script), which compiles the CSS and JS into `public/`.

## Local development

The site is split into two build steps:

* **Assets** (CSS/JS) are compiled by gulp from `assets/`.
* **Pages** (HTML) are generated by `node index.js` from `pages/*.md` and `settings.yaml`.

Running `node index.js` on its own only regenerates the HTML — it does **not** rebuild the CSS/JS. Use the npm scripts so both stay in sync:

| Command | What it does |
|---|---|
| `npm run preview` | Full build (assets + pages), then serve at http://127.0.0.1:8080 |
| `npm run serve` | Serve the existing `public/` folder without rebuilding |
| `npm run build` | Full build (`gulp deploy && node index.js`), no server |
| `npm start` | `node index.js` — regenerate the HTML pages only |

> **Don't open `public/index.html` directly in your browser.** Assets and links use absolute paths (`/css/app.css`, `/about-this-research`), which only resolve when the folder is served over HTTP. Opening the file via `file://` shows an unstyled page with broken links — use `npm run preview` instead.

Override the host/port if 8080 is taken: `PORT=3000 npm run serve`.

## Adding and editing pages

* In `settings.yaml` you can change site-wide settings (title, description, publication date, etc.).

* Most importantly, the `pages:` list defines the title and order of every page. Each entry is an object keyed by `title`:

  ```yaml
  pages:
    - title: "Open APIs for telecoms"
      class: "cover-page"
    - title: "An introduction to Open APIs"
    - title: "About this research"
      appendix: true
  ```

  Per-page keys:

  | Key | Required | Purpose |
  |---|---|---|
  | `title` | yes | Page title; also used to derive the URL slug and the `.md` filename |
  | `class` | no | CSS class added to the page's `<body>`, for page-specific styling |
  | `custom_contents` | no | Custom HTML for this page's entry in the contents list (e.g. a styled label) |
  | `appendix` | no | `true` groups the page under the appendices in the contents list |

* In the `pages/` folder, add a Markdown file for each page. The filename must be a [slugified](https://en.wikipedia.org/wiki/Semantic_URL#Slug) version of the page title, ending in `.md`.

* For example, a page titled `An introduction to Open APIs` needs `pages/an-introduction-to-open-apis.md`.

* The **first** entry in the `pages:` list becomes the cover page, served at the site root (`/`). Its content comes from its title-slug `.md` file like any other page.

* You don't need to add the title inside the Markdown file — it's added automatically from `settings.yaml`.

* If a page goes missing, check that the Markdown filename matches the slugified title in `settings.yaml`. Running `node index.js` logs `[ERROR]` for pages it can't find and `[WARN]` for unused Markdown files.

## Images

### Sizes

Images used in longforms should have a maximum width or height of *1280px* and a standard resolution of *72 dpi*.

### Embedding images

Including images in the GitHub repository makes it sluggish. Instead:

1. Log into the IF CMS.

2. On the sidebar, click "Assets".

3. Click "Longform Assets".

4. Click "Upload files", then choose an image to upload.

5. Find the uploaded image and put a tick next to it.

6. Click the cog icon, then click "View asset".

7. Copy the image URL.

8. In your Markdown document, include an image like this:

```
![This is a description of the image](https://example.com/files/image.png)
```

### Alt tags

These are bits of text that accompany the code to embed images in pages, useful for people that use screen readers. [WebAIM gives this advice](http://webaim.org/techniques/alttext/) for writing alt tags:

* Try to convey the meaning of the image through words
* Don’t write more than a sentence where possible
* Don’t reuse text from the content of a page
* Don’t describe the image as an image
